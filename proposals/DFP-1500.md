## 核心想法

将 `null` 从一个独立的类型降级为指向空对象的**句柄值**，同时引入 `void` 类型用于表示"无返回值"的语义。目标是解决当前 null
和 void 语义混用导致的类型系统混乱。

## 背景与动机

在编写新解析器期间发现以下问题：

1. **语义混用**：`null` 既用作"空值"又用作"无返回值"，导致类型检查时无法区分：
   ```mcdl
   func foo(): null {  // null 作为返回类型？还是返回 null 值？
       return null;
   }
   ```

2. **类型系统不一致**：
    - `null` 既是类型又是值，导致 `typeof(null)` 的语义不明确
    - 无法表达"此函数不返回任何值"的概念（与"返回 null"不同）

3. **缺乏规范**：现有提案中没有明确定义 null 的类型系统位置和行为规范

## 实现路径
引入 void 类型 + null 句柄化

**核心变更**：

- `void` 成为独立类型，仅用于函数返回值
- `null` 从类型降级为 `NullObject` 的单例句柄（类似 Python 的 `None`）
- `null` 的类型是 `NullType`，但 `NullType` 不能作为变量声明类型

**语法示例**：

```mcdl
// 旧代码（将不兼容）
func oldStyle(): null {  // ❌ 编译错误：null 不再是有效的返回类型
    return null;
}

// 新代码
func noReturn(): void {  // ✅ 明确表示无返回值
    print("hello");
    // 隐式 return (无值)
}

func mayReturnNull(): int? {  // ✅ 使用可空类型
    if (condition) return 42;
    return null;  // null 现在是值，不是类型
}

// 类型检查
typeof(null);  // 返回 NullType（但不能用作声明类型）
var x: void;   // ❌ 编译错误：void 只能用于函数返回值
```

**优点**：

- 语义清晰：void = 无返回值，null = 空对象引用
- 符合主流语言习惯（Java、C#、TypeScript）
- 为未来可空类型（`T?`）奠定基础

**缺点**：

- 破坏向后兼容性
- 需要重构大量现有代码
