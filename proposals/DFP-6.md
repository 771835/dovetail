## 摘要

本提案定义 Dovetail 编译器内置注解系统的完整规范，包括注解语法、分类体系、
处理机制、参数规则及各内置注解的语义约定。

---

## 动机

注解系统是 Dovetail 编译器与语言用户之间的重要约定层。当前实现存在以下问题，
需要通过规范加以约束：

1. 注解的合法参数、适用目标、语义边界均无明文规定，编译器行为不可预期
2. 注解处理逻辑与 AST 遍历过度耦合，导致注解语义无法被独立描述和验证
3. 注解对优化器和后端的影响方式不透明，相同注解在不同编译配置下行为不一致
4. 插件系统无法以标准化方式扩展注解

---

## 技术规范

### 一、注解语法

注解仅允许使用内置注解及插件注册的注解，语法如下：

```lark
annotation : "@" ID
           | "@" ID "(" literal ("," literal)* ")"
```

注解必须紧接在其作用目标声明之前。同一目标可携带多个注解，各注解独立处理，
互不影响，除非本规范在特定注解下显式声明冲突规则。

### 二、注解分类

所有内置注解归属于以下五个分类，各分类在编译流程中具有不同的处理方式：

| 分类                 | 说明                        |
|--------------------|---------------------------|
| 条件编译（Condition）    | 在符号构造**前**执行，决定当前声明是否参与编译 |
| 生命周期（Lifecycle）    | 在符号构造**后**执行，标记函数的执行时机    |
| 可见性（Visibility）    | 在符号构造后执行，影响优化器对目标符号的处理策略  |
| 链接（Linkage）        | 在符号构造后执行，控制函数的跨包可见性与调用约定  |
| 元数据（Metadata）      | 附加信息，一般不影响代码生成，供工具链消费     |
| 后端提示（Backend Hint） | 在符号构造后执行，向后端传递优化提示        |

### 三、处理时机

注解的处理时机分为两类，由各注解所属分类决定，不可由用户配置：

**符号构造前（PRE_SYMBOL）**  
在解析当前声明的名称之后、构造符号对象之前执行。此时机下，处理器仅允许
决定是否跳过当前声明，不得向符号写入任何信息。条件编译分类的全部注解
及 `@deprecated` 属于此时机。

**符号构造后（POST_SYMBOL）**  
在符号对象构造完毕、注册到符号表之后执行。此时机下，处理器可读取符号的
完整信息（包括参数类型、返回值类型等），并向符号写入标记（flags）、
元数据（metadata）及覆盖函数类型。其余分类的全部注解属于此时机。

同一声明上的所有注解，每个注解恰好处理一次，不存在重复处理。
两个时机的处理集合互斥，即属于 PRE_SYMBOL 的注解不会在 POST_SYMBOL 阶段被处理，
反之亦然。

### 四、注解对符号的影响

注解处理完毕后，其结果以以下三种方式影响编译流程：

**跳过（skip）**  
仅 PRE_SYMBOL 注解可产生此效果。当结果为跳过时，当前声明的全部内容
（包括参数解析、符号注册、IR 生成）均不执行。

**标记（flags）**  
POST_SYMBOL 注解可向符号附加字符串标记集合。标记是优化器和后端做决策的**唯一重要依据**，
消费方**不应该**仅通过检查注解名称来判断行为。各内置标记的约定语义见 [附表 B](DFP-6-appendix-b.md)。

**元数据（metadata）**  
POST_SYMBOL 注解可向符号附加键值对形式的结构化信息，供后端和工具链按需读取。
各注解的元数据键名约定见 [附表 C](DFP-6-appendix-c.md)。

**函数类型覆盖（type_override）**  
POST_SYMBOL 注解可将函数的类型标记覆盖。

### 五、注解的作用目标限制

部分注解对作用目标有限制，仅允许标注在特定类型的符号上。当注解被用于
不被允许的目标时，编译器必须报告 `AnnotationNotApplicable` 错误并忽略该注解。
各注解的适用目标见 [附表 C](DFP-6-appendix-c.md)。

### 六、实验性注解

部分注解标记为实验性（experimental）。实验性注解在使用时，
编译器必须要求用户显式传入 `--experimental` 编译参数，否则报错并忽略该注解。
当前标记为实验性的注解为 `@extern` 和 `@export`。

### 七、内置注解总览

完整的内置注解列表、参数定义及语义规范见 [附表 C](DFP-6-appendix-c.md)。

### 八、优化器行为约束

优化器在做决策时，必须且只能通过检查符号的 flags 集合来判断是否对某符号执行
特定优化，不得通过注解名称或分类来判断。具体约束如下：

- 若符号的 flags 包含 `no_dce`，则未使用函数消除 pass 不得删除该符号
- 若符号的 flags 包含 `no_inline`，则内联 pass 不得对该符号执行内联
- 若符号的 flags 包含 `aggressive_opt`，则优化器可对该符号执行激进优化策略

### 九、插件扩展

插件推荐通过以下两个步骤向编译器注册自定义注解：

1. 调用 `registry_annotation_spec()` 注入注解的声明，包括名称、参数 schema
   及分类
2. 调用 `registry_annotation_processor_d` 装饰器注册处理器，处理器须继承
   `AnnotationProcessor`基类并指定 `timing` 字段

插件注解的处理时机、参数校验、flags 及 metadata 的语义由插件自行定义，
编译器核心不对其内容作任何假设。
