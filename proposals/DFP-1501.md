## 核心想法

`@extern` 和 `@export` 构成 Dovetail 的 FFI（外部函数接口）双向绑定机制：

- **`@extern`**：声明外部数据包提供的函数接口（导入）
- **`@export`**：标记当前函数供外部数据包调用（导出）

两者都需要通过 `abi` 参数指定调用约定，以确保跨编译器、跨数据包的二进制兼容性。

---

## 背景与动机

### 当前代码状态

从 [builtin_annotations.py](/dovetail/core/builtin_annotation.py) 可见：

```python
"export": Annotation("export", {"path": "", "abi": "dovetail"}, AnnotationCategory.VISIBILITY),
"extern": Annotation("extern", {"path": "", "abi": "dovetail"}, AnnotationCategory.VISIBILITY),
```

### 实际使用场景

**场景 1: 调用 clang-mc 编译的数据包**

```mcdl
// 导入 clang-mc 编译的外部函数
@extern(path="clangmc_lib:math/sqrt", abi="clang-mc")
fn external_sqrt(x: int) -> int;

fn main() {
    let result = external_sqrt(16);  // 调用外部实现
}
```

**场景 2: 导出函数供 clang-mc 调用**

```mcdl
// 导出函数，使用 clang-mc 兼容的 ABI
@export(path="dovetail:get_player_count",abi="clang-mc")
fn get_player_count() -> int {
    return 42;  // 实际实现
}
```

**场景 3: Dovetail 内部互操作(不推荐)**

```mcdl
// 模块 A 导出
@export(path="module_a:utils/shared_utility", abi="dovetail")
fn shared_utility() -> void { ... }

// 模块 B 导入
@extern(path="module_a:utils/shared_utility", abi="dovetail")
fn shared_utility() -> void;
```

## 可能的实现路径

### 路径 1: 最小 ABI 支持（仅 `"clang-mc"` 和 `"dovetail"`）

**设计要点**：

1. **`abi="clang-mc"`**：
    - 生成与 clang-mc 兼容的函数调用约定
    - 限制类型为 FFI-safe 子集（`int`, `bool`, `void`，禁止类实例等复杂特性）

2. **`abi="dovetail"`**：
    - 使用 Dovetail 原生调用约定
    - 支持完整类型系统（包括类、接口、引用）

**类型检查规则**：

```python
FFI_SAFE_TYPES = {"int", "bool", "void", "string"}


def validate_extern_signature(func: FunctionSymbol, abi: str):
    if abi == "clang-mc":
        for param in func.params:
            if param.type.name not in FFI_SAFE_TYPES:
                raise CompileError(f"Type {param.type} is not FFI-safe for clang-mc ABI")
```

**优点**：

- 实现简单，边界清晰
- 强制类型安全，避免运行时崩溃

**缺点**：

- 不支持复杂类型的跨 ABI 传递（如数组、结构体）

---

### 路径 2: 支持 ABI 适配层（未来扩展）

**核心思路**：

- 为 `@extern` 生成"导入桩"（import thunk）
- 为 `@export` 生成"导出桩"（export thunk）
- 桩函数负责参数/返回值的布局转换

**示例（概念性）**：

```mcdl
@extern(path="legacy:process_data", abi="nbt-rpc")
fn process_data(data: MyStruct);

// 编译器生成：
// function __thunk_process_data:
//   1. 将 MyStruct 序列化为 NBT
//   2. 通过 data modify 传递给 legacy:process_data
//   3. 等待回调完成（如果需要）
```

**优点**：

- 支持复杂类型的跨 ABI 互操作
- 隐藏底层序列化细节

**缺点**：

- 性能开销显著
- 增加编译器复杂度
- 当前优先级较低

## 潜在风险

1. **ABI 碎片化**：如果未来引入更多 ABI，可能导致生态分裂
2. **类型不匹配的运行时错误**：编译时无法检测外部函数的实际签名，错误可能延迟到命令执行时
3. **向后兼容性**：修改 `@export` 的参数结构可能破坏现有代码（需要迁移工具）

---

## 实现顺序

1. **Phase 1** (当前草案有效期内)：
    - 为 `@export` 添加 `abi` 参数（默认 `"dovetail"`）
    - 实现 `"clang-mc"` ABI 的类型白名单检查
    - 更新 [DFP 6](/proposals/DFP-6.md) 的注解规范文档
    - 需使用`--experimental`参数启用

2. **Phase 2** (转为正式提案后)：
    - 定义 `path` 参数的完整语法规范
    - 实现链接时的符号签名验证
    - 添加编译器警告：`extern` 函数未找到对应的 `export` 声明
    - 需使用`--experimental`参数启用

3. **Phase 3** (长期特性)：
    - 探索 ABI 适配层的可行性
    - 支持更复杂的类型（如数组、自定义结构体）跨 ABI 传递
    - 默认启用，可通过`--disable-ffi`禁用
