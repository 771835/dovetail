## 摘要

本提案规范 Dovetail 标准库中调试和开发辅助工具的设计原则、接口规范和实现要求，包括日志系统、断言机制和单元测试框架。

---

## 动机

### 问题陈述

1. **缺少结构化日志输出**
    - 开发者依赖手动插入 `tellraw` 命令调试
    - 无法根据环境（开发/生产）控制日志级别
    - 日志格式不统一

2. **缺少运行时检查机制**
    - 函数参数验证依赖返回值约定（易出错）
    - 无法在开发阶段快速暴露逻辑错误
    - 生产环境遗留调试代码影响性能

3. **缺少测试基础设施**
    - 标准库函数正确性无法验证
    - 回归测试困难
    - 缺乏自动化测试能力

### 设计目标

- **统一性**：提供一致的接口和行为
- **灵活性**：支持开发/生产环境的不同需求
- **低开销**：生产环境可完全禁用调试功能
- **可扩展性**：允许用户自定义日志处理器和测试框架

## 技术规范

### 1. 日志系统（`logger.mcdl`）

#### 1.1 日志级别

**定义：**

```mcdl
let LOG_LEVEL_TRACE = 0
let LOG_LEVEL_DEBUG = 1
let LOG_LEVEL_INFO = 2
let LOG_LEVEL_WARN = 3
let LOG_LEVEL_ERROR = 4
let LOG_LEVEL_FATAL = 5
```

**语义：**

- `TRACE`：详细追踪信息（变量值、执行路径）
- `DEBUG`：调试信息（函数调用、状态变化）
- `INFO`：一般信息（系统事件、里程碑）
- `WARN`：警告（非致命错误、降级行为）
- `ERROR`：错误（可恢复的错误）
- `FATAL`：致命错误（不可恢复）

#### 1.2 核心接口

**必需函数：**

```mcdl
/**
 * 核心日志函数
 * 
 * 参数:
 *   level - 日志级别（LOG_LEVEL_* 常量）
 *   tag - 日志标签（模块名、函数名等）
 *   message - 日志消息
 * 
 * 行为:
 *   若 level < 全局日志级别，则忽略该条日志
 *   否则根据配置输出日志（控制台、游戏内、文件等）
 */
fn log_write(level: int, tag: string, message: string)
```

**便捷函数（可选但推荐）：**

```mcdl
fn log_trace(tag: string, message: string)
fn log_debug(tag: string, message: string)
fn log_info(tag: string, message: string)
fn log_warn(tag: string, message: string)
fn log_error(tag: string, message: string)
fn log_fatal(tag: string, message: string)
```

#### 1.3 配置接口

```mcdl
/**
 * 设置全局日志级别
 * 
 * 参数:
 *   level - 新的日志级别
 * 
 * 行为:
 *   修改全局日志级别，影响后续所有 log_write 调用
 */
fn log_set_level(level: int)

/**
 * 获取当前日志级别
 */
fn log_get_level() -> int
```

#### 1.4 输出格式规范

**推荐格式：**

```
[时间戳] [级别] [标签] 消息内容
```

**示例输出：**

```
[12345] [INFO] [PLAYER] Player joined the game
[12346] [WARN] [COMBAT] Damage calculation overflow detected
[12347] [ERROR] [WORLD] Invalid coordinate: y=400
```

#### 1.5 输出目标

实现应支持以下至少一种输出目标：

- **游戏聊天栏**：使用 `tellraw` 命令
- **动作栏**：使用 `title actionbar` 命令
- **记分板**：存储到特定记分板对象（仅计数用途）
- **外部文件**：通过后端支持写入日志文件（可选）

**配置接口（可选）：**

```mcdl
let LOG_TARGET_CHAT = 0
let LOG_TARGET_ACTIONBAR = 1
let LOG_TARGET_SCOREBOARD = 2

fn log_set_target(target: int)
```

#### 1.6 颜色编码（Minecraft 特定）

实现应为不同日志级别提供颜色区分（可选但推荐）：

| 级别    | 推荐颜色 | Minecraft 颜色代码 |
|-------|------|----------------|
| TRACE | 灰色   | `gray`         |
| DEBUG | 白色   | `white`        |
| INFO  | 绿色   | `green`        |
| WARN  | 黄色   | `yellow`       |
| ERROR | 红色   | `red`          |
| FATAL | 深红   | `dark_red`     |

### 2. 断言系统（`assert.mcdl`）

#### 2.1 设计原则

1. **开发/生产分离**：断言仅在开发环境启用
2. **快速失败**：断言失败应立即报告并可选中断执行
3. **描述性错误**：断言失败应提供清晰的错误信息

#### 2.2 核心接口

**基础断言：**

```mcdl
/**
 * 断言某个条件为真
 * 
 * 参数:
 *   condition - 待检查的条件
 *   message - 断言失败时的错误消息
 * 
 * 行为:
 *   若 ASSERT_ENABLED 为 false，立即返回（无操作）
 *   若 condition 为 false，记录错误并根据配置采取行动：
 *     - 输出错误消息到日志（级别 ERROR 或 FATAL）
 *     - 增加全局断言失败计数器
 *     - 可选：触发游戏内警告（红色文本、音效等）
 *     - 可选：中断当前函数执行（需语言支持）
 */
fn assert(condition: boolean, message: string)
```

**专用断言（可选但推荐）：**

```mcdl
/**
 * 断言两个值相等
 * 
 * 参数:
 *   actual - 实际值
 *   expected - 期望值
 *   message - 基础错误消息
 * 
 * 行为:
 *   若 actual != expected，生成详细错误消息：
 *   "{message} (expected: {expected}, actual: {actual})"
 */
fn assert_eq(actual: int, expected: int, message: string)

/**
 * 断言两个值不相等
 */
fn assert_ne(actual: int, not_expected: int, message: string)

/**
 * 断言值在指定范围内
 */
fn assert_in_range(value: int, min: int, max: int, var_name: string)

/**
 * 断言值为真（布尔）
 */
fn assert_true(value: boolean, message: string)

/**
 * 断言值为假（布尔）
 */
fn assert_false(value: boolean, message: string)
```

#### 2.3 配置接口

```mcdl
/**
 * 全局断言开关（建议编译时常量）
 */
let ASSERT_ENABLED = true  // 或通过编译参数控制

/**
 * 断言失败时是否中断执行（可选特性）
 */
let ASSERT_BREAK_ON_FAIL = false

/**
 * 获取断言失败次数
 */
fn assert_get_fail_count() -> int

/**
 * 重置断言失败计数
 */
fn assert_reset_count()
```

#### 2.4 实现要求

1. **性能约束**：
    - 当 `ASSERT_ENABLED = false` 时，断言调用应编译为空操作（或由编译器优化移除）
    - 条件表达式应在检查前求值（避免副作用）

2. **错误报告**：
    - 断言失败应通过日志系统输出（使用 `log_error` 或 `log_fatal`）
    - 应包含文件名、行号（若编译器支持）

3. **集成要求**：
    - 断言模块应依赖日志模块
    - 断言失败计数器应持久化（使用记分板或全局变量）

### 3. 单元测试框架（`unittest.mcdl`）

#### 3.1 设计原则

1. **简洁性**：API 简单易用，降低测试编写门槛
2. **独立性**：每个测试用例应独立执行，互不影响
3. **报告性**：清晰报告测试结果（通过/失败/总数）

#### 3.2 核心接口

**测试套件管理：**

```mcdl
/**
 * 开始一个测试套件
 * 
 * 参数:
 *   suite_name - 测试套件名称
 * 
 * 行为:
 *   - 输出测试套件开始标记
 *   - 重置测试统计计数器（总数、通过、失败）
 */
fn test_suite_begin(suite_name: string)

/**
 * 结束测试套件并输出报告
 * 
 * 行为:
 *   - 输出测试统计（总数、通过、失败）
 *   - 输出总体结果（全部通过或有失败）
 *   - 返回失败测试数量（可选）
 */
fn test_suite_end() -> int  // 返回失败数
```

**测试用例：**

```mcdl
/**
 * 执行相等性测试
 * 
 * 参数:
 *   test_name - 测试名称（描述性字符串）
 *   expected - 期望值
 *   actual - 实际值
 * 
 * 行为:
 *   - 比较 expected 和 actual
 *   - 若相等：增加通过计数，输出成功标记（可选静默）
 *   - 若不等：增加失败计数，输出详细错误信息
 */
fn test_assert_eq(test_name: string, expected: int, actual: int)

/**
 * 执行不等性测试
 */
fn test_assert_ne(test_name: string, not_expected: int, actual: int)

/**
 * 执行布尔测试
 */
fn test_assert_true(test_name: string, condition: boolean)
fn test_assert_false(test_name: string, condition: boolean)

/**
 * 执行范围测试
 */
fn test_assert_in_range(test_name: string, value: int, min: int, max: int)
```

#### 3.3 测试统计接口

```mcdl
/**
 * 获取当前测试统计
 */
fn test_get_total_count() -> int
fn test_get_passed_count() -> int
fn test_get_failed_count() -> int
```

#### 3.4 输出格式规范

**测试套件开始：**

```
========== Test Suite: mathlib ==========
```

**测试结果（可选详细输出）：**

```
✓ abs(5) == 5
✓ abs(-5) == 5
✗ gcd(12, 8) == 4 - Expected: 4, Got: 3
```

**测试套件结束：**

```
========== Results ==========
Total: 10
Passed: 9
Failed: 1
```

#### 3.5 实现要求

1. **集成要求**：
    - 测试框架应依赖日志模块和断言模块
    - 测试失败应使用 `log_error` 记录

2. **可扩展性**：
    - 允许用户定义自定义测试断言函数
    - 支持测试套件嵌套（可选高级特性）

3. **性能考虑**：
    - 测试代码应可编译时排除（通过编译参数）
    - 测试函数应使用特定命名约定（如 `test_*` 前缀）

## 使用规范

### 4.1 日志系统使用示例

```mcdl
include "logger"

fn process_player_action(player: string, action: string) {
    log_debug("PLAYER", f"Processing action: {action} for {player}")
    
    if (action == "jump") {
        log_trace("PLAYER", "Jump action detected")
        // 处理跳跃
    } else {
        log_warn("PLAYER", f"Unknown action: {action}")
    }
}

@init
fn main() {
    // 开发环境：显示所有日志
    log_set_level(LOG_LEVEL_TRACE)
    
    // 生产环境：仅显示警告和错误
    // log_set_level(LOG_LEVEL_WARN)
    
    log_info("MAIN", "Datapack initialized")
}
```

### 4.2 断言系统使用示例

```mcdl
include "assert"
include "mathlib"

fn safe_divide(a: int, b: int) -> int {
    assert_ne(b, 0, "Division by zero")
    return a / b
}

fn calculate_stats(health: int, max_health: int) {
    assert_in_range(health, 0, max_health, "health")
    assert(max_health > 0, "max_health must be positive")
    
    let percent = (health * 100) / max_health
    assert_in_range(percent, 0, 100, "calculated percent")
    
    return percent
}
```

### 4.3 单元测试使用示例

```mcdl
include "unittest"
include "mathlib"

fn test_mathlib_basic() {
    test_suite_begin("mathlib_basic")
    
    test_assert_eq("abs(5)", 5, abs(5))
    test_assert_eq("abs(-5)", 5, abs(-5))
    test_assert_eq("abs(0)", 0, abs(0))
    
    test_assert_eq("max(3, 7)", 7, max(3, 7))
    test_assert_eq("max(7, 3)", 7, max(7, 3))
    test_assert_eq("max(5, 5)", 5, max(5, 5))
    
    test_assert_eq("min(3, 7)", 3, min(3, 7))
    
    test_suite_end()
}

fn test_mathlib_advanced() {
    test_suite_begin("mathlib_advanced")
    
    test_assert_eq("gcd(12, 8)", 4, gcd(12, 8))
    test_assert_eq("gcd(17, 5)", 1, gcd(17, 5))
    test_assert_eq("lcm(4, 6)", 12, lcm(4, 6))
    
    test_assert_eq("pow(2, 10)", 1024, pow(2, 10))
    test_assert_eq("pow(3, 3)", 27, pow(3, 3))
    test_assert_eq("pow(5, 0)", 1, pow(5, 0))
    
    test_suite_end()
}

@init
fn run_all_tests() {
    log_set_level(LOG_LEVEL_INFO)
    
    test_mathlib_basic()
    test_mathlib_advanced()
    
    log_info("TEST", "All test suites completed")
}
```

## 编译器支持要求

### 5.1 必需特性

本提案的实现**不依赖**以下编译器特性：

- ✅ 基本函数定义和调用
- ✅ 全局变量（用于配置和状态）
- ✅ 字符串格式化（`f"..."`）
- ✅ 条件语句（`if`）

### 5.2 可选优化

编译器**可选**提供以下优化：

1. **常量折叠**：
   ```mcdl
   if (ASSERT_ENABLED == false) {
       assert(x > 0, "...")  // 编译时移除整个语句
   }
   ```

2. **死代码消除**：
   ```mcdl
   // 若 LOG_LEVEL = INFO，以下调用应被移除
   log_trace("TAG", "message")
   log_debug("TAG", "message")
   ```

3. **内联展开**：
   ```mcdl
   // log_info 可内联为直接调用 log_write
   fn log_info(tag: string, msg: string) {
       log_write(LOG_LEVEL_INFO, tag, msg)
   }
   ```

4. **测试代码剥离**：
    - 通过编译参数排除 `test_*` 函数
    - 移除 `unittest.mcdl` 模块的依赖

### 5.3 调试信息支持（可选）

编译器**可选**提供以下元数据用于调试：

```mcdl
// 伪代码：编译器插入文件名和行号
assert(condition, f"[{__FILE__}:{__LINE__}] {message}")
```

## 实现建议

### 6.1 最小实现

最简可用版本应包含：

1. **logger.mcdl**：
    - 5 个日志级别常量
    - `log_write` 核心函数
    - `log_set_level` 配置函数
    - 使用 `tellraw` 输出到聊天栏

2. **assert.mcdl**：
    - `ASSERT_ENABLED` 常量
    - `assert` 基础函数
    - `assert_eq` 相等性断言
    - 依赖 `logger.mcdl`

3. **unittest.mcdl**：
    - `test_suite_begin/end` 套件管理
    - `test_assert_eq` 测试断言
    - 依赖 `logger.mcdl` 和 `assert.mcdl`

### 6.2 扩展实现

完整版本可额外包含：

1. **logger.mcdl**：
    - 6 个便捷函数（`log_trace` 等）
    - 颜色编码支持
    - 多输出目标（动作栏、记分板）
    - 时间戳支持（使用游戏刻）

2. **assert.mcdl**：
    - 5 个专用断言（`assert_ne`, `assert_in_range` 等）
    - 断言失败计数器
    - 详细错误格式化

3. **unittest.mcdl**：
    - 4 个测试断言变体
    - 测试统计接口
    - 彩色输出（通过或失败）

## 版本兼容性

### 7.1 向后兼容

- 新增函数应保持接口稳定
- 配置常量的默认值不应随意更改
- 日志格式可迭代，但应保持基本结构

### 7.2 废弃策略

若需移除函数，应：

1. 在至少一个版本中标记为 `@deprecated`
2. 提供迁移路径（替代函数）
3. 在文档中说明废弃原因
