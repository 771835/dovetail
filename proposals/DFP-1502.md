## 核心想法

在**类型系统层面**废止现有 Array 类型及其一些机制，引入直接映射到 Minecraft 原生存储的 `list`、`dict` 等复合类型。通过包装类型
`ref[T]`
区分"值类型"（直接存储数据）和"引用类型"（存储引用地址），形成类似 C++ 值对象与堆对象语义的类型层次。

## 背景与动机

### 现有类型系统分析

当前类型系统（[dovetail/core/enums/types.py](/dovetail/core/enums/types.py)）包含：

1. **基础类型**：DataType 枚举（int, string, boolean, null）
2. **数组类型**：Array 类，支持 `int[10]`、`int[]`、`int[10][20]` 等形式
3. **结构体类型**：Structure 类，用于用户定义的复合类型

**问题**：

- Array 类型（[types.py#L106-L145](/dovetail/core/enums/types.py)）是一种抽象，未直接映射到 MC 原生存储
- 缺少映射到 NBT List/Compound 的内建类型
- 类型层面无法区分"值语义"和"引用语义"

### Minecraft 原生存储语义

| 存储位置             | 支持类型                                        | 语义特点         |
|------------------|---------------------------------------------|--------------|
| **Scoreboard**   | int                                         | 值类型，数据直接存储   |
| **NBT List**     | byte/short/int/long/float/double/string 的列表 | 值类型，数据嵌入 NBT |
| **NBT Compound** | 键值对                                         | 值类型，数据嵌入 NBT |

### 符号与类型的关系

符号系统（[dovetail/core/symbols/](/dovetail/core/symbols)）中的 Variable、Function 等符号通过 `dtype` 字段引用类型。重构类型系统后：

```python
# 符号引用类型，类型决定存储语义
variable = Variable(name="nums", dtype=ListType(DataType.INT))
variable = Variable(name="nums_ref", dtype=RefType(ListType(DataType.INT)))
```

## 可能的实现路径

### 路径1：类型层次设计（推荐）

#### 类型系统架构

```python
# 基础类型 - 映射到 Scoreboard 或 NBT 基本类型
class PrimitiveType(DataTypeBase):
    """原始类型"""
    INT = 'int'  # Scoreboard int
    FLOAT = 'float'  # NBT float/double
    STRING = 'string'  # NBT string
    BOOLEAN = 'boolean'  # NBT byte (0/1)


# 复合值类型 - 数据直接嵌入 NBT
class ListType(DataTypeBase):
    """列表类型，映射到 NBT List
    
    示例: list[int] -> NBT List [I;1,2,3]
    """

    def __init__(self, element_type: DataTypeBase):
        self.element_type = element_type

    def get_name(self) -> str:
        return f"list[{self.element_type.get_name()}]"


class DictType(DataTypeBase):
    """字典类型，映射到 NBT Compound
    
    示例: dict[string, int] -> NBT Compound {key1:1, key2:2}
    """

    def __init__(self, key_type: DataTypeBase, value_type: DataTypeBase):
        self.key_type = key_type
        self.value_type = value_type

    def get_name(self) -> str:
        return f"dict[{self.key_type.get_name()},{self.value_type.get_name()}]"


class TupleType(DataTypeBase):
    """元组类型，固定长度的异构列表
    
    示例: tuple[int, string] -> NBT List [I:1, S:"hello"]
    """

    def __init__(self, element_types: list[DataTypeBase]):
        self.element_types = element_types

    def get_name(self) -> str:
        types_str = ','.join(t.get_name() for t in self.element_types)
        return f"tuple[{types_str}]"


# 引用包装类型 - 存储引用地址而非数据
class RefType(DataTypeBase):
    """引用包装类型
    
    区分:
    - list[int]: 值类型，数据直接存储在变量位置（如 NBT List）
    - ref[list[int]]: 引用类型，存储引用地址，实际数据在 Storage/Entity
    
    类似 C++ 的栈分配 vs 堆分配语义
    """

    def __init__(self, target_type: DataTypeBase):
        self.target_type = target_type

    def get_name(self) -> str:
        return f"ref[{self.target_type.get_name()}]"
```

#### 存储语义映射表

| 类型声明                | 存储位置       | MC 映射                  | 语义   |
|---------------------|------------|------------------------|------|
| `int`               | Scoreboard | objective: score       | 值类型  |
| `string`            | Storage    | Storage string tag     | 值类型  |
| `list[int]`         | Storage    | Storage List[I;...]    | 值类型  |
| `dict[string, int]` | Storage    | Storage Compound {...} | 值类型  |
| `ref[int]`          | 全局对象池      | 对象句柄                   | 引用类型 |
| `ref[list[int]]`    | 全局对象池      | 对象句柄                   | 引用类型 |

#### 语法示例

```dovetail
// ========== 值类型示例 ==========

// 基础值类型
let x: int = 42
let flag: boolean = true
let name: string = "player"

// 复合值类型 - 数据直接嵌入
let nums: list[int] = [1, 2, 3]           // NBT: [I;1,2,3]
let names: list[string] = ["a", "b", "c"] // NBT: ["a","b","c"]
let player_stats: dict[string, int] = {  // NBT: {health:100, mana:50}
    "health": 100,
    "mana": 50
}

// 元组类型
let pos: tuple[int, int, int] = (10, 20, 30) // NBT: [I;10,20,30]

// ========== 引用类型示例 ==========

// 引用类型 - 存储引用地址
let global_data: ref[dict[string, int]] = Storage.load("player_data")
let shared_list: ref[list[int]] = Storage.load("inventory")

// 引用的引用
let ref_ref: ref[ref[list[int]]] = global_data

```

**优点**：

- 类型系统清晰，直接映射到 MC 原生能力
- 编译期即可确定存储位置和优化策略
- 支持灵活的嵌套类型

**缺点**：

- 需要大规模重构类型系统
- 引用类型的所有权管理复杂度较高

### 路径2：通过类型属性而非包装类型区分

使用类型属性标注存储位置，而非包装类型：

```python
class TypeKind(Enum):
    VALUE = "value"  # 值类型，直接存储数据
    STORAGE_REF = "storage_ref"  # 引用类型，指向 Storage


class DataTypeBase:
    kind: TypeKind = TypeKind.VALUE
    # ... 其他字段


class ListType(DataTypeBase):
    def __init__(self, element_type: DataTypeBase, kind: TypeKind = TypeKind.VALUE):
        self.element_type = element_type
        self.kind = kind
```

语法：

```dovetail
let nums: list<int, storage> = Storage.load("my_list")  // kind=storage_ref
let local_nums: list<int, value> = [1,2,3]            // kind=value (默认)
```

**优点**：

- 避免嵌套类型（ref[list[int]]）
- 类型表达式更简洁

**缺点**：

- 语法不够直观，需要记忆类型属性
- 扩展性较差，难以表达复杂引用关系

### 路径3：只实现基础的值存储

仅实现路径1中值存储部分，ref机制交由标准库或用户自行实现，尽量减少编译器内部职权

语法：
```dovetail
let nums: list<int> = Storage.load("my_list")
let local_nums: list<int> = [1,2,3]
```

## 待解决的问题

- [ ] **类型推导规则**：如何从字面量 `[1,2,3]` 推导出 `list[int]`？
- [ ] **引用所有权**：`ref[T]` 的生命周期管理是否需要类似 Rust 的借用检查？
- [ ] **NBT 性能优化**：频繁的 list/dict 操作如何减少 `/data get` 调用？
- [ ] **向后兼容**：现有 `int[10]` 语法如何迁移？
- [ ] **类型相等性**：`ref[list[int]]` 和 `list[int]` 是否可互转？

## 潜在风险

1. **破坏性变更**：现有使用 Array 类型的代码需要重写
2. **类型复杂度**：开发者需要理解值类型与引用类型的语义差异
3. **性能差异**：NBT 访问比 Scoreboard 慢，需要优化策略
4. **调试难度**：引用类型的生命周期管理可能引入难以追踪的 bug
