## 摘要

本提案定义 Dovetail 的插件系统架构、注册机制、接口规范和生命周期管理。插件系统允许第三方开发者通过注册后端、优化器等组件来扩展编译器功能，而无需修改编译器核心代码。

---

## 动机

### 问题陈述

1. **编译器功能扩展困难**
    - 新增后端需要修改核心代码
    - 优化器无法独立开发和测试
    - 功能模块与编译器紧耦合

2. **缺少标准化扩展机制**
    - 第三方开发者无规范可循
    - 不同扩展之间无法互操作
    - 版本兼容性难以保证

3. **职责边界不清晰**
    - 编译器核心功能边界模糊
    - 无法灵活组合不同工具链组件

### 设计目标

- **模块化**：编译器核心最小化，通过插件注册扩展功能
- **统一接口**：所有扩展通过 `plugin_api` 统一接入
- **灵活性**：单个插件可注册多种类型的组件
- **可组合性**：插件之间可以互相依赖和协作

## 核心概念

### 1. 插件与组件

**插件（Plugin）**：

- 插件是扩展的容器，负责向编译器注册各种组件
- 单个插件可以注册多个后端、多个优化器等
- 插件本身不限定类型

**组件（Component）**：

- 组件是具体的功能实现，如后端、优化器、构建工具等
- 组件由插件在初始化时注册到编译器
- 编译器通过组件管理器调度各组件

**关系**：

```
插件 (Plugin)
  ├─ 注册后端 A (Backend)
  ├─ 注册后端 B (Backend)
  ├─ 注册优化器 X (IROptimizationPass)
  └─ 注册优化器 Y (IROptimizationPass)
```

### 2. Plugin API

**Plugin API** 是编译器提供的核心接口层：

- 所有插件通过 `plugin_api` 与编译器交互
- `plugin_api` 本身也以插件形式实现，保证扩展性
- 提供统一的注册、配置、事件等服务

### 3. 插件类型

| 类型        | 说明                                                    |
|-----------|-------------------------------------------------------|
| `plugin`  | 普通插件                                                  |
| `library` | 依赖库插件，用于提供接口供其他插件调用，此类库原则上**不应该**影响编译流程               |
| `loader`  | 插件加载器，**仅**存在**一个**且插件目录名为 `plugin_loader`，与编译器共享命名空间 |

## 技术规范

### 1. 插件发现与加载

#### 1.1 插件位置

插件**仅**存放在编译器安装目录下：

```
<dovetail_install_dir>/
└── plugins/
    ├── plugin_official_backends/     # 官方后端插件
    ├── plugin_core_optimizers/       # 核心优化器插件
    ├── plugin_api/                   # Plugin API 插件
    └── plugin_loader/                # 插件加载器
```

**设计原则**：

- **全局唯一**：避免路径搜索的复杂性和冲突
- **集中管理**：便于版本控制和依赖解析
- **权限隔离**：插件与用户代码分离

#### 1.2 插件目录结构

**最小结构**：

```
<plugin_name>/
└── plugin.toml          # 必需：插件元数据
```

**约束**：

- 插件目录名**必须**与 `plugin.toml` 中的 `plugin.name` 字段相同
- 不强制要求 `__init__.py`
- 实际入口由 `plugin.toml` 中 `[plugin].entry` 指定
- 单个插件可包含多个模块文件

---

### 2. 插件元数据

#### 2.1 `plugin.toml` 格式

**示例**：

```toml
[plugin]
name = "official_backends"
version = "1.0.5"
api_version = "2.3"
entry = "loader"
type = "plugin"

[metadata]
author = "Yours Team"
description = "Official backend collection"
license = "MIT"
homepage = "https://example.com"

[dependencies]
plugin_core_optimizers = "^1.0.0"

[compatibility]
dovetail_min = "0.5.0"
dovetail_max = "1.0.0"

[config]
backend_default = "je1214"
enable_experimental = false
```

**字段规范**：

| 字段                   | 类型     | 必需 | 说明                                    |
|----------------------|--------|----|---------------------------------------|
| `plugin.name`        | String | 是  | 插件唯一标识符，**必须与目录名相同**（小写字母、数字、下划线）     |
| `plugin.version`     | String | 是  | 插件版本（语义化版本，格式：`主版本.次版本.修订版本`）         |
| `plugin.api_version` | String | 是  | 依赖的 Plugin API 版本（格式：`主版本.次版本`，不含修订号） |
| `plugin.entry`       | String | 是  | 入口模块名（不含 `.py` 扩展名）                   |
| `plugin.type`        | String | 是  | 插件类型（`plugin` / `library` / `loader`） |
| `metadata.*`         | String | 否  | 插件元信息                                 |
| `dependencies.*`     | String | 否  | 依赖的其他插件及版本约束（语义化版本范围）                 |
| `compatibility.*`    | String | 否  | 编译器版本兼容性约束                            |
| `config.*`           | Any    | 否  | 插件默认配置                                |

#### 2.2 入口类规范

**约束**：

- 入口模块**必须**包含名为 `PluginMain` 的类
- `PluginMain` **必须**继承自 `plugin_api.Plugin`
- `PluginMain` **必须**实现 `load()`、`unload()`、`validate()` 方法

**接口定义**：

```python
class PluginMain(Plugin):
    def load(self) -> None:
        """加载插件，注册组件"""

    def unload(self) -> bool:
        """卸载插件，返回是否成功"""

    def validate(self) -> tuple[bool, str | None]:
        """验证插件环境，返回 (是否通过, 错误信息)"""
```

**职责**：

- `load()`：注册后端、优化器、事件回调等组件
- `unload()`：清理资源，返回 `False` 表示拒绝卸载
- `validate()`：验证运行环境（依赖库、配置等），返回 `(False, "错误原因")` 阻止加载

### 3. Plugin API 规范

#### 3.1 核心接口

Plugin API 提供以下服务：

**插件基类**：

```python
class Plugin(ABC):
    def load(self) -> None: ...

    def unload(self) -> bool: ...

    def validate(self) -> tuple[bool, str | None]: ...

    def initialize(self) -> None: ...

    def get_memory_usage(self) -> int: ...

    def handle_message(self, sender: Plugin, message: Any) -> Any: ...

    def send_message(self, target: str | Plugin, message: Any) -> bool: ...
```

**组件注册接口**：

```python
# 注册后端
def register_backend(self, backend_class: type[Backend]) -> None: ...


# 注册优化 Pass
def register_optimize_pass(self, pass_class: type[IROptimizationPass]) -> None: ...


# 注册内置库
def register_library(self, name: str, lib: type[Library]) -> None: ...


# 注册编译器注解
def register_annotation(self, name: str, annotation: Annotation) -> bool: ...
```

**配置接口**：

```python
# 获取配置（自动读取当前插件配置）
def get_config(self, key: str, default: Any = None, plugin: str | None = None) -> Any: ...


# 设置配置
def set_config(self, key: str, value: Any) -> None: ...


# 编译器信息
@property
def compiler_version(self) -> str: ...


@property
def install_dir(self) -> Path: ...


@property
def api_version(self) -> str: ...
```

**事件接口**：

```python
# 订阅编译器事件
def register_event(self, event_name: str, callback: Callable) -> None: ...


# 取消订阅
def unregister_event(self, event_name: str, callback: Callable) -> None: ...
```

#### 3.2 Plugin API 加载

**Plugin API 本身是插件**，位于：

```
plugins/
└── plugin_api/
    ├── plugin.toml
    └── main.py
```

**plugin.toml 内容**：

```toml
[plugin]
name = "plugin_api"
version = "2.3.0"
api_version = "2.3"
entry = "main"
type = "plugin"
```

**特殊说明**：

- `plugin_api` 提供 `Plugin` 基类供所有插件继承
- `plugin_loader` 也继承自 `plugin_api.Plugin`
- `plugin_loader` 与编译器共享同一命名空间，其他插件在独立命名空间中运行

### 4. 后端注册机制

#### 4.1 后端接口

后端**必须**继承自 `Backend` 基类并实现以下接口：

```python
class Backend(ABC):
    def __init__(self, ir_builder: IRBuilder, target: Path, config: CompileConfig): ...

    @staticmethod
    @abstractmethod
    def is_support(config: CompileConfig) -> bool: ...

    @staticmethod
    @abstractmethod
    def get_name() -> str: ...

    def generate(self) -> None: ...
```

**约束**：

- `get_name()` 返回后端唯一标识符
- `is_support()` 判断当前配置是否支持该后端
- `generate()` 执行代码生成

#### 4.2 后端注册

**注册方式**：

```python
class PluginMain(Plugin):
    def load(self):
        self.register_backend(JE1214Backend)
        self.register_backend(BedrockBackend)
```

**行为规范**：

- 同名后端注册时，后注册的覆盖先注册的
- 后端名称区分大小写
- 编译器应记录后端注册来源（哪个插件）

### 5. 优化管道注册机制

#### 5.1 优化器接口

**优化 Pass 接口**：

```python
class IROptimizationPass(ABC):
    def __init__(self, builder: IRBuilder, config: CompileConfig): ...

    @classmethod
    def get_metadata(cls) -> PassMetadata: ...

    @abstractmethod
    def execute(self) -> bool: ...

    def should_run(self, context: OptimizationContext) -> bool: ...

    def analyze(self) -> dict: ...
```

**Pass 元数据**：

```python
@dataclass
class PassMetadata:
    name: str  # Pass 唯一标识
    display_name: str  # 显示名称
    description: str  # 功能描述
    level: OptimizationLevel  # 最低优化级别
    phase: PassPhase  # 执行阶段
    required_features: tuple[str, ...]  # 必需特性
    provided_features: tuple[str, ...]  # 提供特性
    incompatible_with: tuple[str, ...]  # 不兼容的 Pass
```

**约束**：

- `execute()` 返回 `True` 表示 IR 发生变更
- `should_run()` 根据上下文判断是否执行
- `get_metadata()` 返回 Pass 元数据，用于依赖解析和调度

#### 5.2 优化 Pass 注册

**注册方式**：

```python
@register_pass(PassMetadata(
    name="constant_folding",
    display_name="常量折叠",
    description="在编译时计算常量表达式",
    level=OptimizationLevel.O1,
    phase=PassPhase.TRANSFORM,
    provided_features=("simplified_arithmetic",)
))
class ConstantFoldingPass(IROptimizationPass):
    def execute(self) -> bool: ...


class PluginMain(Plugin):
    def load(self):
        self.register_optimize_pass(ConstantFoldingPass)
```

**说明**：

- `Optimizer` 负责管理和执行 `IROptimizationPass`
- 编译器根据优化级别、依赖关系和 Pass 元数据调度执行
- Pass 执行顺序由拓扑排序和阶段（Phase）决定

### 6. 事件系统

#### 6.1 标准事件

| 事件名称            | 触发时机   | 传递参数                               |
|-----------------|--------|------------------------------------|
| `pre_parse`     | 解析文件前  | `file_path: Path`                  |
| `post_parse`    | 解析文件后  | `ast: AST`                         |
| `pre_semantic`  | 语义分析前  | `ast: AST`                         |
| `post_semantic` | 语义分析后  | `ast: AST, symbols: SymbolTable`   |
| `pre_ir_gen`    | IR 生成前 | `ast: AST`                         |
| `post_ir_gen`   | IR 生成后 | `ir: IR`                           |
| `pre_optimize`  | 优化前    | `ir: IR, level: OptimizationLevel` |
| `post_optimize` | 优化后    | `ir: IR`                           |
| `pre_codegen`   | 代码生成前  | `ir: IR, backend: Backend`         |
| `post_codegen`  | 代码生成后  | `result: BackendResult`            |

#### 6.2 事件注册

**注册方式**：

```python
class PluginMain(Plugin):
    def load(self):
        self.register_event('post_parse', self.on_parse_complete)

    def on_parse_complete(self, ast):
        self.logger.info(f"Parsed {ast.node_count()} nodes")
```

**行为规范**：

- 同一事件的多个回调按注册顺序调用
- 回调**不应**修改传入的数据结构（除非明确允许）
- 回调抛出异常时，编译器应记录错误并继续执行

### 7. 插件生命周期

#### 7.1 加载顺序

```
编译器启动
└─> 加载 plugin_loader（硬编码，共享命名空间）
   └─> plugin_loader 扫描插件目录
      └─> 加载 plugin_api
         └─> plugin_api 初始化核心服务
      └─> 读取其他 plugin.toml
         └─> 解析依赖关系，拓扑排序
         └─> 按依赖顺序加载插件（独立命名空间）
            └─> 导入入口模块
            └─> 实例化 PluginMain
            └─> 调用 validate()
            └─> 调用 load()
               └─> 编译器就绪
```

**关键约束**：

- `plugin_loader` **必须**先于所有其他插件加载
- `plugin_api` **必须**先于业务插件加载
- 被依赖的插件**必须**先于依赖者加载
- 循环依赖**必须**被检测并报错

#### 7.2 插件依赖

**依赖声明**：

```toml
[dependencies]
plugin_core_optimizers = "^1.0.0"
plugin_api = "^2.3.0"
```

**依赖解析规则**：

- 版本约束遵循语义化版本规范
- 缺失依赖**必须**阻止插件加载并报错
- 版本不兼容**必须**阻止插件加载并报错

**错误处理**：

```
error: unresolved plugin dependency
  --> plugin_name/plugin.toml:8:1
   |
 8 | plugin_core_optimizers = "^1.0.0"
   | ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ plugin not found
   |
   = help: install the dependency with: dovetail plugin install plugin_core_optimizers
```

### 8. 版本兼容性

#### 8.1 Plugin API 版本

**格式**：`主版本.次版本`（不含修订号）

**兼容性规则**：

- 插件的 `api_version` 主版本**必须**与当前 Plugin API 主版本相同
- 插件的 `api_version` 次版本**不得**高于当前 Plugin API 次版本

**示例**：

```
当前 Plugin API 版本：2.3

✓ 兼容：2.0, 2.1, 2.2, 2.3
✗ 不兼容：1.x, 2.4, 3.x
```

#### 8.2 编译器版本约束

**约束声明**：

```toml
[compatibility]
dovetail_min = "0.5.0"
dovetail_max = "1.0.0"
```

**检查规则**：

- 编译器版本 < `dovetail_min`：跳过插件，记录警告
- 编译器版本 > `dovetail_max`：跳过插件，记录警告
- 未指定约束：与所有版本兼容

### 10. 性能考虑

#### 10.1 性能预算

**约束**：

- 单个优化 Pass 开销 < 编译时间的 10%
- 所有优化管道开销 < 编译时间的 50%
- 后端生成时间与 IR 长度成线性关系
- 事件回调延迟 < 50ms

#### 10.2 内存管理

**建议**：

- 插件应及时释放不再使用的资源
- 避免在全局变量中缓存大型数据
- 使用弱引用避免循环引用
- 实现 `get_memory_usage()` 方法供编译器监控

---

### 11. 插件隔离

**约束**：

- 除 `plugin_loader` 外，所有插件在独立命名空间中运行
- 插件原则上**不应**直接访问其他插件的内部状态
- 插件间通信**推荐**通过 `send_message()` / `handle_message()` 进行

### B. 参考实现

**最小插件示例**：

```
my_plugin/
├── plugin.toml
└── main.py
```

**plugin.toml**：

```toml
[plugin]
name = "my_plugin"
version = "1.0.0"
api_version = "2.3"
entry = "main"
type = "plugin"
```

**main.py**：

```python
from dovetail.plugins.plugin_api import Plugin


class PluginMain(Plugin):
    def load(self):
        self.logger.info("Plugin loaded")

    def unload(self) -> bool:
        return True

    def validate(self) -> tuple[bool, str | None]:
        return True, None
```
